# Story 6.4: Add Optional Auto-Punctuation and Sentence-Aware Reading Groups

Status: done

## Story

As Clint,
I want the app to infer punctuation and sentence/clause grouping for Chinese text that arrives without punctuation,
so that the reading result is easier to follow, wraps more naturally, and spoken playback includes more natural pauses.

## Acceptance Criteria

1. **Given** OCR and pinyin succeed for Chinese text that contains little or no punctuation, **When** the result is displayed, **Then** the API may return a separate `data.reading` projection with derived punctuated reading groups **And** the raw `data.ocr` and `data.pinyin` payloads remain unchanged for diagnostics and fallback use.

2. **Given** a reading projection is present, **When** the UI renders the primary reading view, **Then** it uses `reading.groups` as the default reading surface **And** each reading group references canonical `pinyin.segments` via `segment_indexes` **And** no duplicated ruby-ready token payload is introduced into the API contract.

3. **Given** a reading group spans more than one raw pinyin segment, **When** the backend builds `reading.groups`, **Then** the group may combine multiple adjacent raw segments **And** grouping is limited to adjacent segments with the same `line_id` **And** the original segment order is preserved.

4. **Given** auto-punctuation was applied, **When** line-level or page-level pronunciation playback is triggered, **Then** playback uses the derived `playback_text` from the reading group where available **And** the UI can communicate that auto-punctuation was applied through explicit provider metadata.

5. **Given** punctuation inference is disabled, unavailable, fails, or produces no useful grouping improvement, **When** the result is displayed, **Then** the app falls back to the current line-group rendering and playback behavior **And** no OCR or pinyin result data is lost.

6. **Given** the reading projection is generated by different provider types over time, **When** the API returns `data.reading`, **Then** the provider boundary is explicit through metadata that can represent `heuristic`, `remote_service`, or `llm` **And** the contract remains stable as the implementation evolves.

7. **Given** all existing backend and frontend tests run, **When** story 6.4 is implemented, **Then** current raw-segment contracts remain backward compatible **And** the new behavior is covered by targeted tests for fallback, grouping, and playback text selection.

## Dev Notes

### Story Source and Business Context

Epic 6 already delivered translation and browser-native pronunciation controls, but the approved sprint change proposal identified a remaining gap: the product currently preserves OCR lines, not assisted reading sentences. For punctuationless Chinese pages, that is good enough for fidelity but not for scanability, natural playback pauses, or wrap behavior.

This story exists to add an assistive reading layer on top of the existing raw pipeline. It is not a rewrite of OCR, pinyin generation, or diagnostics.

### Core Product Constraints

- The raw OCR and pinyin payloads remain the system of record.
- `data.reading` is additive and optional.
- Derived reading output is for the main reading surface and playback text selection, not for replacing the underlying evidence trail.
- If the derived layer is unavailable, low-value, or wrong, the current grouped-line rendering must remain usable without loss of data.

### Current Brownfield Reality

The implementation must follow the current codebase, not the aspirational component map in the architecture:

- Result rendering, grouped-line logic, line/page playback state, and fallback messaging all still live in [frontend/src/features/process/components/UploadForm.jsx](/Users/clint/Documents/GitHub/ocr-pinyin/frontend/src/features/process/components/UploadForm.jsx)
- Backend request orchestration for `/v1/process` lives in [backend/app/api/v1/process.py](/Users/clint/Documents/GitHub/ocr-pinyin/backend/app/api/v1/process.py)
- Pinyin generation is one `PinyinSegment` per OCR segment today in [backend/app/services/pinyin_service.py](/Users/clint/Documents/GitHub/ocr-pinyin/backend/app/services/pinyin_service.py)
- Translation enrichment already runs after pinyin generation and groups by `line_id` in [backend/app/services/translation_service.py](/Users/clint/Documents/GitHub/ocr-pinyin/backend/app/services/translation_service.py)

That means the safest insertion point for this story is:

1. OCR
2. Pinyin generation
3. Translation enrichment
4. Derived reading projection
5. Frontend rendering/playback preference for `reading.groups`

Do not push punctuation inference down into OCR normalization or the pinyin provider.

### Approved Schema Direction

Use a separate reading projection under `ProcessData` with explicit provider metadata.

Recommended shape:

```json
{
  "data": {
    "ocr": { "segments": [] },
    "pinyin": { "segments": [] },
    "reading": {
      "mode": "derived",
      "provider": {
        "kind": "heuristic",
        "name": "built_in_rules",
        "version": "v1",
        "applied": true,
        "confidence": 0.78,
        "request_id": null,
        "warnings": []
      },
      "groups": [
        {
          "group_id": "rg_0",
          "line_id": 0,
          "raw_text": "老师叫同学们好我们开始上课了",
          "display_text": "老师叫同学们好，我们开始上课了。",
          "playback_text": "老师叫同学们好，我们开始上课了。",
          "confidence": 0.78,
          "segment_indexes": [0, 1, 2]
        }
      ]
    }
  }
}
```

Locked rules from the approved proposal and epic:

- no mutation of raw `ocr` or `pinyin` payloads
- no duplicated ruby-ready token structures in the API
- groups may span multiple adjacent raw segments, but only within the same `line_id`
- no cross-line grouping and no segment reordering
- provider metadata must explicitly support `heuristic`, `remote_service`, or `llm`

### Frontend Guardrails

Current playback behavior in `UploadForm.jsx` already assumes:

- `groupSegmentsByLine(...)` derives grouped reading rows from `pinyin.segments`
- `buildSpokenLineText(...)` currently concatenates raw `source_text`
- there is exactly one active utterance at a time
- page playback and line playback are mutually exclusive

This story should reuse that orchestration rather than replacing it with a new playback stack.

Preferred approach:

1. Introduce a derived reading-group model in the frontend that can resolve canonical `pinyin.segments` through `segment_indexes`
2. Render ruby from canonical segments, not from duplicated reading payload tokens
3. Choose `playback_text` when a derived reading group is active
4. Fall back to the current grouped-line logic when `reading` is absent or not applied

Do not break the quiet result presentation with banners, toasts, or noisy explanations. If punctuation was inferred, communicate it with a small note near the reading content, not a disruptive alert.

### Backend Guardrails

Current backend contracts and tests already enforce stable envelopes:

- [backend/app/schemas/process.py](/Users/clint/Documents/GitHub/ocr-pinyin/backend/app/schemas/process.py)
- [backend/tests/unit/schemas/test_process_response_contract.py](/Users/clint/Documents/GitHub/ocr-pinyin/backend/tests/unit/schemas/test_process_response_contract.py)
- [backend/tests/integration/api_v1/test_process_route.py](/Users/clint/Documents/GitHub/ocr-pinyin/backend/tests/integration/api_v1/test_process_route.py)

This story must preserve:

- success responses still require `data` and `diagnostics`
- partial responses still require `data`, `warnings`, and `diagnostics`
- error responses still exclude `data`, `warnings`, and `diagnostics`
- existing clients continue to work when `reading` is absent

The new reading service should be best-effort. If it cannot improve the result safely, it should degrade quietly without contaminating the raw pipeline or turning a successful OCR+pinyin response into an error.
