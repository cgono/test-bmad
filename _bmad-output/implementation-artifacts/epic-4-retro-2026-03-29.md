## Challenges

### 1. In-Memory Cost Store Is a Known Fragility

`DailyCostStore` (Story 4.4) resets on process restart, same as `MetricsStore` from Epic 3. At personal/MVP scale this is acceptable, but a Render instance restart mid-day silently zeros daily cost tracking. No test will catch this — it's a runtime behavior gap.

### 2. Two Sprint Changes Mid-Epic — Worth Distinguishing the Types

Both proposals executed cleanly, but they represent different root causes:
- **2026-03-28 restructuring** (renumbered 4 stories): spec gap — scope wasn't fully defined at epic start
- **2026-03-29 versioning/render**: legitimate discovery — staging + release management needs only became apparent after first production deployment

Sprint changes from real-world discovery are healthy. Sprint changes from under-specified scope are worth reducing.

### 3. Staging Environment Was a Mid-Epic Spec Gap

The need for a staging environment wasn't anticipated at epic planning. It surfaced mid-sprint and was handled via the change proposal process, but the staging spec remains incomplete. Operational concerns (environments, release management, cost observability) tend to surface *after* first deployment — harder to prevent than code spec gaps, but worth naming.

## Lessons Learned

1. **Incremental, dependency-ordered story design scales well for feature clusters.** Stories 4.3–4.6 are the clearest example yet — each left the next a clean foundation with no reach-around or undo required. Replicate this sequencing for Epic 5's history/storage work, which has a similar dependency chain (5.1 → 5.2 → 5.3 before 5.4).

2. **In-memory state is a recurring deferral, not a resolved issue.** `MetricsStore` (Epic 3) → `DailyCostStore` (Epic 4). Both are deliberate choices. Epic 5 introduces explicit persistence — make a concrete decision at Story 5.1 spec time whether cost tracking piggybacks.

3. **Operational infrastructure concerns (environments, release management) surface post-deployment, not pre.** Staging environment need appeared after first production usage. Build in a prompt "what operational gaps might appear once this is live?" during epic planning for infrastructure-heavy epics.

4. **The sprint change proposal process is mature enough to be trusted.** It handled two mid-epic scope events without disruption, and produced implementation artifacts for Story 4.7 as a side effect. It is now part of the standard workflow, not an exception handler.
