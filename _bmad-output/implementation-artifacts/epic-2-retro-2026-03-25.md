## What Went Well

### 1. Spec-Driven Development — Zero First-Pass Failures

Every story in Epic 2 was implemented without debugging sessions, reverts, or blockers. The Dev Notes sections provided exact code snippets, file-by-file changelists, and explicit "do NOT touch" lists. All five stories implemented cleanly on the first pass. This is a direct payoff from Epic 1's architectural groundwork and the story creation discipline developed there.

### 2. Error Taxonomy Scaled Without Breaking Changes

The `{category}_{specific_condition}` naming convention from Epic 1 was extended smoothly. Three new codes added:
- `ocr_no_chinese_text` (Story 2.1)
- `reason_code="pinyin_execution_failed"` on `PinyinSegment` (Story 2.2)
- `ocr_low_confidence` (Story 2.4)

None required taxonomy changes or introduced naming conflicts.

### 3. Partial Response Pattern Evolved Coherently

Story 2.3 introduced "pinyin fails → partial without pinyin." Story 2.4 added "low confidence → partial WITH pinyin." Two distinct partial states, handled correctly by the same schema validator (`data is not None`). The design accommodated both without backtracking.

### 4. Zero Reverts — Epic 1 Lesson Applied

Epic 1 had a reverted commit (wrong-scope Story 1.6 attempt). Epic 2: zero. The sprint change discipline held.

### 5. Clean Architecture Layering Maintained

Providers stayed thin. Service layer owned business logic. Route handler orchestrated without inspecting internals. The `api / schemas / services / adapters / core / middleware` split was respected across all 5 stories, including the cross-layer Stories 2.3 and 2.4 that touched both backend and frontend.

## Challenges

### 1. Ruff Lint Errors Surfaced Late and Remain Unresolved

Story 2.4 noted: *"Pre-existing ruff E501/I001/E402 errors in `schemas/process.py` and `test_process_response_contract.py` not introduced by this story."*

These errors exist in the codebase. They were noticed in Story 2.4 but described as "not introduced by this story" — meaning they accumulated somewhere during the Epic 2 arc without triggering CI failures. Both files will be modified in Epic 3 (Story 3.1 adds diagnostics structure), which means they need to be cleaned up before then.

**Root cause:** CI ruff enforcement may have gaps that allow lint violations to pass silently.

### 2. Tech Writer Prep Task Carried Forward Again

"Document the sprint change proposal process" was a nice-to-have from Epic 1's retro. It had no story file, no explicit owner, and no trigger. It didn't happen in Epic 2. Tasks without story files tend to drift.

---

## Lessons Learned

1. **The detailed story spec pattern is working — maintain it.** Epic 2's zero-blocker execution is directly attributable to story specs that include exact implementation guidance. Don't trade this for speed.

2. **Watch for lint debt accumulating silently.** CI passed, but ruff errors were in the codebase by Story 2.4. Tighten the CI gate so violations block rather than pass through.

3. **Unowned prep tasks don't ship.** "Document sprint change proposal process" has now crossed two epics without landing. If it matters, it needs a story file and an owner. If it doesn't matter, remove it.
