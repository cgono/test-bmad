## Challenges

### 1. render.yaml Required Four Post-Merge Fixes

```
d54209b feat: pin PYTHON_VERSION and UV_VERSION
793203b fix: remove "free" plan from static frontend
2583eb3 fix: set render.yaml services to "free"
759c3c6 fix: move Python version in render.yaml
```

**Root cause:** The spec was built on outdated Render documentation. Free tier plan syntax, Python version placement, and static service plan handling had all changed from the references used. The config was internally consistent but incorrect against the actual current platform state. No contract verification step existed to catch this before merge.

**Key insight from Clint:** "We should be verifying the contracts with external dependencies. The 4 commits were due to outdated info informing the render.yaml file." The fix is a process addition — not more careful spec writing, but a verification step that checks current provider docs before the spec is written.

### 2. "Document Sprint Change Proposal Process" — Three Epics, Still Unshipped

This task has appeared in three consecutive retrospectives. Root cause is consistent across all three: no story file, no explicit owner, no trigger. Floating action items without story files do not ship.

---

## Lessons Learned

1. **Infrastructure stories need a pre-implementation contract verification step.** Before writing a spec for external platform configuration (Render, cloud providers, third-party APIs), explicitly verify current provider docs. Do not rely on cached references or prior examples. Add this to the definition-of-done for infrastructure stories: "current provider documentation reviewed and config validated against actual platform."

2. **If a task has survived three epics without a story file, it needs one now or it should be closed.** The sprint change proposal documentation crossed three epics as a floating action item. A story file with acceptance criteria is what makes it shippable. Floating action items are not accountable.

3. **Code review on schema-heavy foundational stories is a quality multiplier.** Story 3.1 established the diagnostics schema contract used by all downstream stories and the frontend. It entered review with five correctness gaps. All were caught. For stories that establish foundational schemas or API contracts, code review is a required quality gate.
