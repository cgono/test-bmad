## What Went Well

### 1. The epic stayed additive instead of disruptive

Story 6.1 added `translation_text` to `PinyinSegment` while explicitly preserving existing response envelopes, diagnostics contracts, and `/v1/process` behavior. Stories 6.2 and 6.3 then delivered pronunciation entirely in the frontend, which kept later scope changes from rippling back through backend contracts.

### 2. The team reused one strong UI abstraction repeatedly

Epic 6 consistently built on the grouped-line model introduced earlier in Story 4.2:
- Story 6.1 used `line_id` grouping for one translation row per rendered line
- Story 6.2 used the same grouping to build line-level spoken text
- Story 6.3 used the same grouping again for top-to-bottom page playback

This is the clearest sign of a good intermediate abstraction: later stories got meaningfully better without reworking the underlying shape.

### 3. Scope was tightened to fit the real codebase

The story notes repeatedly corrected for brownfield reality: the result UI still lives in `frontend/src/features/process/components/UploadForm.jsx`, not in the more aspirational frontend structure from planning docs. That prevented architecture drift and kept implementation honest.

### 4. Non-blocking degradation rules were handled well

Translation failure was treated as non-fatal in Story 6.1. Browser speech support, missing voices, and speech errors were treated as local UI limitations in Stories 6.2 and 6.3. Across all three stories, the primary reading result remained usable even when the enhancement path failed.

### 5. Review feedback was applied instead of ignored

Stories 6.2 and 6.3 both contain concrete review findings that were patched before closeout. That kept the new playback behavior from shipping with obvious state-management bugs and improved the test suite around interruption and page/line switching.

## Challenges

### 1. Epic completion state and story reality drifted apart

The story files for 6.1, 6.2, and 6.3 are all `done`, but `sprint-status.yaml` still marks `epic-6` as `in-progress`. That weakens the retrospective trigger, makes progress reporting less trustworthy, and leaves ambiguity about whether more Epic 6 scope is expected.

### 2. Playback logic concentrated risk in one large frontend file

Stories 6.2 and 6.3 both expanded `UploadForm.jsx`. The work was successful, but the deferred items show a recurring pattern: session guards, closure sensitivity, render-order-derived keys, and browser-event races are all being managed in one component-level state machine. That is workable now, but it is the main fragility cluster in the epic.

### 3. Test confidence was asymmetrical across the epic

Story 6.1 validated backend and frontend paths. Stories 6.2 and 6.3 were intentionally frontend-only, but Story 6.3 still records an unresolved backend regression validation gap caused by the local backend test environment. Even when changes are frontend-only, repeated inability to run the backend suite reduces confidence in the overall system state.

### 4. Story inference worked, but it also exposed planning looseness

Stories 6.2 and 6.3 were inferred from the reserved "future audio pronunciation" scope rather than being explicitly authored in the original epic plan. The team adapted well, but this is still a planning smell: the outcome was good because the scope was narrow and the developer had enough context, not because the epic was fully specified.

---

## Lessons Learned

1. **A stable presentation model enables cheap feature stacking.** `line_id` grouping unlocked translation, line playback, and page playback with little contract churn. Preserve and extend this pattern rather than bypassing it.

2. **Frontend-only extensions are the right default when the user value is immediate and persistence is unnecessary.** Browser speech synthesis delivered pronunciation without dragging in storage, media generation, or new backend APIs.

3. **Brownfield correction notes in story files are paying for themselves.** Calling out the difference between planning docs and the live codebase kept implementation practical and avoided invented structure.

4. **Deferred review items are starting to cluster around playback orchestration rather than product behavior.** That is a signal to simplify the speech state model before adding richer pronunciation features, not after.

5. **Epic priority changes need a stronger closure ritual.** Reordering Epic 6 ahead of Epic 5 worked, but it also made "previous epic" and "next epic" reasoning less obvious. The tracker and retro flow need to reflect non-linear epic order more explicitly.
